# Project Name

Мобильное приложение для генерации музыки.
Стек: **Flutter (mobile) + Go API + Go Worker + Redis + Postgres + S3**.

Проект сделан как **монорепозиторий**, чтобы:

* backend и mobile развивались синхронно
* один запуск поднимал всё окружение
* не было рассинхроена контрактов API

# cmd/

```
cmd/api
cmd/worker
```

По Go-практике **каждый бинарь имеет свою папку в cmd/**.

### cmd/api

Точка входа HTTP сервера.

Здесь только:

* загрузка конфига
* сборка зависимостей
* запуск сервера

❗ Бизнес-логики здесь быть не должно.

---

### cmd/worker

Отдельный процесс для фоновых задач.

Почему отдельно:

* тяжёлые операции не блокируют HTTP
* можно масштабировать независимо
* соответствует async-архитектуре (jobs → queue → worker)

---

# internal/

Весь приватный код приложения.

Особенность Go:

> всё внутри internal нельзя импортировать снаружи

Это **защита архитектуры на уровне компилятора**.

Если не использовать internal — проект быстро превращается в "общую помойку" из пакетов.

---

## internal/config

Загрузка переменных окружения.

Зачем:

* единая точка конфигурации

---

## internal/httpapi

HTTP инфраструктура:

* server
* router
* middleware
* handlers

Отделяем это от main.go, чтобы:

* код был тестируемым
* не раздувать main

---

## internal/httpapi/handlers/health.go

```
GET /health
```

Первый и обязательный endpoint.

Нужен для:

* docker/k8s health checks
* мониторинга
* быстрого теста “сервер жив”

## internal/worker

Здесь будет:

* запись статусов
